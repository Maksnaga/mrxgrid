# Tutoriel — 8. Gotchas (annexe)

> Index des pièges connus rencontrés sur la lib + son écosystème
> (Vue 3, Mozaic, virtualisation). Référence rapide pour quand "ça
> marche pas comme prévu".

## Vue 3

### Slot scope qui mange ses props

`<template #edit-brand>` sans destructuring → tes props arrivent
`undefined`. **Toujours destructurer** :

```vue
<!-- ❌ Faux -->
<template #edit-brand>
  <BrandEditor :field="…" />  <!-- field = undefined -->
</template>

<!-- ✅ Bon -->
<template #edit-brand="{ field, rowIndex, editValue, updateValue, commit }">
  <BrandEditor :field="field" :model-value="editValue" @commit="commit('down')" />
</template>
```

### Reactivity sur les rows

Mutating `row[field] = x` côté grid (`applyFills`) ne propage pas aux
computeds qui ne lisent que la **référence d'array** (`filteredRows`
quand aucun filter actif). La lib bump un `dataVersion` interne dans
`applyFills` pour invalider la cascade. **Si tu mutes manuellement
en dehors de la lib, fais le bump à la main :**

```ts
import { useGridState } from '@adeo/grid'

const state = useGridState()
row.field = newValue
state.dataVersion.value++   // ← force invalidation
```

### `storeToRefs` vs `toRefs`

Pinia + destructuring direct → perte de réactivité :

```ts
// ❌ Perd la réactivité
const { rows, total } = useProductsStore()

// ✅ Garde la réactivité
const { rows, total } = storeToRefs(useProductsStore())
```

### `markRaw` sur les renderers

Sans `markRaw`, Vue wrap le composant en reactive() → identity
breakdown, re-render à chaque accès :

```ts
// ❌ Triggers warnings + re-renders
export const YesNoRenderer = defineComponent({ … })

// ✅ Inerte côté réactivité
export const YesNoRenderer = markRaw(defineComponent({ … }))
```

## Mozaic UI components

### `MSelect` retourne des strings

`MSelect` sérialise tout en string dans son `<option value>`. Si tes
options ont des booleans (`value: true`), tu reçois `"true"` au commit.
**Coerce systématiquement** :

```ts
function coerceYesNo(v: unknown): true | false | null {
  if (v === true || v === 'true') return true
  if (v === false || v === 'false') return false
  return null
}
```

### `MDrawer.close-on-overlay` fire sur le body

Par défaut `:close-on-overlay="true"` ferme aussi sur les clicks
**dans le dialog body** (clics sur whitespace entre les items). Le
grid passe `false` sur ses drawers internes. **Fais pareil sur tes
drawers** :

```vue
<MDrawer :open="open" :close-on-overlay="false" title="…">
  <!-- contenu -->
</MDrawer>
```

L'utilisateur ferme via le bouton × ou le bouton "Apply" du footer.

### `MDatepicker` inutilisable en cellule contrainte

Quand tu mets `MDatepicker` dans une cellule étroite (~150 × 40 px),
les pseudo-éléments `::-webkit-datetime-edit-*` se stackent verticalement
à cause de `display: flex` sur le wrapper. **Utilise `<input type="date">`
natif** dans les éditeurs de cellule. La lib le fait par défaut quand
tu mets `cellEditor: 'date'`.

## adeo-grid — comportements à connaître

### Le label header s'affiche avec ellipsis quand serré

À partir de la version où `min-width: 120 px` est enforcé sur le
resize, le label header est `white-space: nowrap; text-overflow:
ellipsis`. Tu vois `PROD…` au lieu de "Produit" si la colonne fait
moins de ~150 px (en gros tous les chars du label ne tiennent pas).
**C'est intentionnel** — la sort indicator + kebab restent visibles.
Le label complet reste accessible via hover (tooltip à venir).

### Resize ne peut pas descendre sous 120 px

Hard floor sur le drag resize. **Pour aller plus bas**, déclare
`ColumnDef.minWidth: '40px'` (par exemple pour une colonne d'actions
icon-only). Ton minWidth gagne contre le default.

### `maxWidth` sur les longs textes

L'autosize au mount mesure avec canvas. Une description multi-lignes
peut donner 1500 px → tasse les autres colonnes. **Mets toujours
`maxWidth: '320px'` (ou similaire) sur les colonnes texte long.**

### Un seul pinned 'start' à la fois

Plusieurs `pinned: 'start'` colonnes écrasent la zone center. Garde un
seul (généralement l'identifiant principal) pour qu'il fasse office de
"row header".

### Auto-size all → seuil à 1000 samples

Le sampling est uniforme : `stride = rows / SAMPLE_CAP`. Le cap est à
1000 (auparavant 5000). Si ta donnée a un outlier extrême en row #237
qui n'est pas dans le sample, la colonne ne le voit pas. **L'utilisateur
peut toujours drag-resize manuellement.**

### Resize → trigger sort

Bug classique : drag du resize handle, mouseup → browser fire un
synthetic click sur le header → tri se déclenche. La lib gère ça via
un flag `wasResizingRecently()` qui ignore le click pendant 200 ms
après mouseup. **Si tu vois encore le bug, mets ta version à jour.**

### scrollCellIntoView au click — pas de scroll si visible

Click sur une cell **déjà visible** ne déclenche **pas** de scroll. Le
trigger est strict : `cellTop < scrollTop` (caché derrière le sticky
header) ou `cellBottom > viewportBottom`. La marge de confort ne padde
que la **position cible** du scroll, pas la condition.

### `:loading` vs `:refreshing`

- `loading: true` → squelette plein écran. Use cases : premier load,
  page change, sort change, filter change, search.
- `refreshing: true` → aucun visuel par défaut. Active le slot
  `#loading` que tu peux remplir. Use case : refresh silencieux
  post-mutation.

**Bug typique** : tu mets `loading: true` à chaque refetch (même
post-mutation) → squelette plein écran clignote à chaque cell-edit.
**Règle** : `silent: true` après mutation, `silent: false` après input
utilisateur.

### Export sur 1M rows = crash Chrome

L'export sérialise tout en mémoire. ~300 MB sur 1M rows. **Toujours
scope `selection` ou `visible`** sur les gros datasets, ou stream
côté serveur. La lib défaut à `selection` quand il y en a une.

### Bulk emit threshold à 1000

Au-delà de 1000 cellules modifiées en une opération (Ctrl+A → Delete,
paste range), la lib émet `bulkCellEdit` au lieu de `cellEdit`
per-cell. **Si tu wires seulement `cellEdit`, le clear Ctrl+A ne fait
rien côté API.** Wire `bulkCellEdit` aussi.

## Patterns à NE PAS faire

### ❌ Muter `props.rows` directement

```ts
// ❌ Vue va warn, et la prop reste figée côté parent
props.rows.push(newProduct)
```

À la place, le parent gère son state et passe une nouvelle référence.

### ❌ Importer toute la lib Mozaic dans `<AdeoGrid>`

La lib importe ce dont elle a besoin (MTag pour le renderer 'tag',
MDrawer pour les drawers internes). **Ne ré-importe pas** Mozaic dans
ton wrapper — tu doubles le bundle.

### ❌ Reactive() sur le payload du JSON

```ts
// ❌ Inutile — `ref()` deep-wrappe déjà
const data = reactive(jsonData)

// ✅ Suffisant
const data = ref(jsonData)
```

### ❌ Recompute des columns à chaque render

```ts
// ❌ Re-build à chaque keypress dans la searchbox
const columns = payload.meta.fields.map(buildColumn)

// ✅ Computed = cache la dernière valeur
const columns = computed(() => payload.meta.fields.map(buildColumn))
```
