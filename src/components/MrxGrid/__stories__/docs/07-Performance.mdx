# Performance

> Comment garder le grid à 60 fps même sur 100k lignes — et les pièges à éviter.

## Limites observées

    <tr><th>Métrique</th><th>Limite confortable</th><th>Limite ressentie</th></tr>

## Patterns à suivre

<div className="do">
✅ <strong>Utilise <code>:row-id</code> pour les datasets non triviaux.</strong> Sans
identifiant stable, la sélection / les formules / l'expansion sont indexées
par numéro de ligne et cassent au moindre re-tri.
</div>

```vue
<MrxGrid :rows="rows" :row-id="(row) => String(row.id)" />
```

<div className="do">
✅ <strong>Active <code>:virtual-scroll</code> dès 1 000 lignes.</strong> Le coût fixe
(sizer + observers) est payé une fois ; le gain au scroll est immense.
</div>

<div className="do">
✅ <strong>Pinne les colonnes critiques aux deux extrémités.</strong> Quand
<code>:virtual-columns</code> est activé, les pinned (start/end) sont toujours
rendus — utile pour ID / actions.
</div>

<div className="do">
✅ <strong>Pour les cellules custom, utilise <code>markRaw()</code> sur les renderers.</strong>
Les components Vue proxy-és par défaut ; <code>markRaw()</code> évite l'overhead du
reactive system pour des shapes statiques.
</div>

```ts
import { markRaw, defineComponent } from 'vue'

const StarRating = markRaw(defineComponent({ /* ... */ }))
{ field: 'rating', renderer: StarRating }
```

<div className="do">
✅ <strong>Stocke <code>columns</code> en dehors du composant si possible.</strong> Une
réaffectation de l'array <code>columns</code> à chaque render fait recalculer le
pipeline. Définis-le en module-level si statique.
</div>

<div className="do">
✅ <strong>Lazy-load la donnée en pages.</strong> 50 000 lignes ne devraient pas être
fetch d'un coup. Combine <code>:total-count</code> + <code>:on-visible-range-change</code>.
</div>

<div className="do">
✅ <strong>Active <code>:server-filter</code> + <code>:server-grouping</code> quand le
backend peut le faire.</strong> Le grid filtre / groupe en mémoire par défaut, ce
qui est rapide jusqu'à ~ 50 000 lignes mais bloque l'UI au-delà.
</div>

## Patterns à éviter

<div className="dont">
❌ <strong>Ne mute pas <code>rows</code> en place</strong> sans réaffecter le ref. Vue ne
détecte pas les mutations profondes. Préfère immutable :
</div>

```ts
// ❌ pas tracké
rows.value[3].name = 'New'

// ✅ tracké
rows.value = rows.value.map((r, i) => i === 3 ? { ...r, name: 'New' } : r)
```

<div className="dont">
❌ <strong>Ne crée pas un new <code>ColumnDef[]</code> à chaque render.</strong> La référence
d'array change → tout le pipeline recalcule.
</div>

```ts
// ❌
<MrxGrid :columns="rows.map(_ => ({ field: 'x', headerName: 'X' }))" />

// ✅
const columns = computed<ColumnDef[]>(() => [
  { field: 'x', headerName: 'X' },
  // …
])
```

<div className="dont">
❌ <strong>Ne mets pas le grid dans un <code>v-for</code> avec re-mount fréquent.</strong>
Chaque mount allocate les composables, observers, listeners. Un seul grid
utilise 1-3 MB de heap.
</div>

<div className="dont">
❌ <strong>Ne fais pas d'IO dans <code>valueFormatter</code> / <code>valueGetter</code>.</strong>
Ils sont appelés à chaque render de cellule (potentiellement 10 fois / seconde
au scroll). Pre-compute en amont.
</div>

<div className="dont">
❌ <strong>N'utilise pas de gros slots <code>#cell</code> avec subcomponents lourds</strong>
sans <code>shallowRef</code> / <code>markRaw</code>. Le slot est instancié pour chaque cellule
visible (jusqu'à 80 × 30).
</div>

## Diagnostic — qu'est-ce qui freine ?

### Symptôme : scroll saccadé

1. Ouvre Chrome DevTools → Performance → Record un scroll
2. Cherche les long tasks (&gt; 16 ms)
3. Si c'est dans <em>Scripting</em> → renderer custom trop lourd
4. Si c'est dans <em>Layout</em> → trop de cellules rendues (cap dépassé ?)
5. Si c'est dans <em>Painting</em> → CSS coûteux (filters, box-shadow, blend modes)

### Symptôme : freeze au tri / filtre

Le pipeline est synchrone. Pour 100 000+ lignes, un tri full-array peut
prendre 100-300 ms. Solutions :

- Active <code>:server-side</code> et délègue au backend
- Ou : pré-trie les données côté serveur, désactive le tri client
- Ou : utilise un Web Worker (non livré, mais le moteur est pure-functions)

### Symptôme : clic sur cellule met &gt; 100 ms à répondre

Probablement un re-render global. Vérifie que :
- Le parent ne rebind pas <code>:columns</code> ou <code>:rows</code> à chaque event
- Pas de <code>watch</code> qui fait un effet de bord lourd dans le consommateur
