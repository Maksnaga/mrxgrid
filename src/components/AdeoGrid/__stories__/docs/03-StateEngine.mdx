# State & Engine

> Comment l'état est centralisé, qui peut le lire, qui peut l'écrire, et
> pourquoi cette séparation est essentielle.

## Le pattern central

L'idée : **un seul `useGridState()` par grid**, partagé via `provide()`. Toutes
les features lisent / écrivent à travers cet état. Aucune ne possède son propre
état global. Ça permet :

- L'undo/redo cohérent (un seul state à snapshotter)
- La persistance localStorage en un seul tour
- Les plugins lisent l'état sans drilling
- Les tests unitaires : on stub `useGridState` et on teste les engines isolément

## `useGridEngine<T>(state)` — les opérations

L'engine **n'a pas de state** propre — il prend le state en paramètre et expose
des fonctions qui le lisent/écrivent. Tu peux donc instancier autant d'engines
que tu veux sur le même state.

```ts
const state  = useGridState<Product>()
const engine = useGridEngine<Product>(state)

engine.sort.toggleSort('price', /* multi: */ false)   // single-sort par prix
engine.sort.clearSort()
engine.filter.setModel({ conditions: [...] }, 'replace')
engine.group.addField('category')
engine.formula.set({ rowId: '42', field: 'total' }, '=[qty]*[price]')
engine.selection.selectAll()
engine.clipboard.copy()      // copie la sélection courante
engine.history.undo()
```

## Persistance

Avec `:persist-key="catalogue-v1"`, le grid sauve **automatiquement** dans
`localStorage["catalogue-v1"]` à chaque changement de :

- largeur de colonne
- ordre des colonnes
- visibilité (hidden / shown)
- pinning (left / right / null)
- multi-sort stack
- filter model du builder

Et restaure tout au prochain mount. Aucun appel impératif requis. Si tu veux
piloter à la main, le grid expose `persistView()` / `restoreView()` via
`defineExpose`.

## Plugins

Un plugin reçoit `{ state, engine }` à `init()` et peut renvoyer un cleanup.
Idéal pour : audit log, telemetry, keybindings custom, intégration analytics —
tout ce qui ne concerne pas le rendu.

```ts
import type { MrxGridPlugin } from '@/components/AdeoGrid'

const auditPlugin: MrxGridPlugin = {
  id: 'audit',
  init({ engine }) {
    const off = engine.events?.on('cellEditCommit', (e) => {
      myTelemetry.send('cell-edit', { field: e.field, row: e.rowIndex })
    })
    return () => off?.()
  },
}

// usage
<MrxGrid :columns="cols" :rows="rows" :plugins="[auditPlugin]" />
```

## Pourquoi pas Pinia ?

Pinia est installé dans le projet mais n'est pas utilisé par le grid : un grid
est une feature **locale** par instance (chaque grid a son propre état). Pinia
sert pour le state global de l'app, pas pour les composants composables. Le
pattern `useGridState()` + `provide/inject` est plus naturel et n'introduit
aucune dépendance forte côté consommateur.
