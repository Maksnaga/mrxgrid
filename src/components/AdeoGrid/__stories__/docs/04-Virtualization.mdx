# Virtualization

> Comment afficher 100&nbsp;000 lignes × 200 colonnes sans tuer le DOM.

## Principe

Le DOM est lent : 1 000 nœuds = 1 frame, 100 000 nœuds = 30 secondes de freeze.
La solution : **ne rendre que ce qui est visible**. MrxGrid implémente la
virtualisation sur deux axes indépendants :

- `useVirtualScroll` — vertical, sur les **lignes**
- `useVirtualColumns` — horizontal, sur les **colonnes**

Composés ensemble par `useVirtualGrid`, qui n'expose qu'une slice 2D.

## Vertical — comment ça marche

<div className="ascii">
{`Wrapper            (overflow: auto, fixed height H)
┌────────────────────────────────────┐  ▲
│ ╔════════════════════════════════╗ │  │
│ ║       Sticky header            ║ │  │
│ ╚════════════════════════════════╝ │  │ scroll =
│                                    │  │ wrapper.scrollTop
│ Body sizer (height = N × rowH)     │  │
│ ┌────────────────────────────────┐ │  │
│ │  Top spacer (offsetY px)       │ │  ▼
│ ├────────────────────────────────┤
│ │  Row[start]                    │  ← rendu
│ │  Row[start+1]                  │  ← rendu
│ │   …                            │
│ │  Row[end]                      │  ← rendu
│ ├────────────────────────────────┤
│ │  Bottom space (sizer fills)    │
│ └────────────────────────────────┘
└────────────────────────────────────┘

  rowsRendered = end − start  (capé à 80, voir plus bas)
  start        = floor(scrollTop / rowHeight) − overscan
  end          = ceil((scrollTop + H) / rowHeight) + overscan
  offsetY      = start × rowHeight  ← top spacer height`}
</div>

### Détails clés

<table>
  <thead>
    <tr><th>Décision</th><th>Raison</th></tr>
  </thead>
  <tbody>
    <tr>
      <td><strong>Index-based, pas array-based</strong></td>
      <td><code>visibleRange</code> est un <code>[start, end]</code> int. Le rendu fait <code>v-for="i in renderRange"</code> + <code>getRow(i)</code>. Pas d'allocation d'array <code>.slice()</code> à chaque scroll → 0 GC pressure.</td>
    </tr>
    <tr>
      <td><strong>Top spacer en height, pas en transform</strong></td>
      <td><code>translateY()</code> crée un containing block qui piège <code>position: sticky</code>. Les colonnes pinned cesseraient d'être collées. On utilise un <code>div</code> de hauteur <code>offsetY</code> à la place.</td>
    </tr>
    <tr>
      <td><strong>Cap à 80 lignes max</strong></td>
      <td>En densité compact (25px), un viewport 600px contient 24 lignes. Avec overscan 5 de chaque côté, on rend 34. Le cap protège contre des resize qui voudraient rendre 200+ lignes (Mac touchpad inertia).</td>
    </tr>
    <tr>
      <td><strong>scrollTop clampé quand le contenu rétrécit</strong></td>
      <td>Si l'utilisateur scrolle à 80 000 puis active le groupage qui réduit à 4 lignes, on doit forcer <code>scrollTop = 0</code> sinon le browser garde la valeur invalide.</td>
    </tr>
    <tr>
      <td><strong>rowHeight reactif</strong></td>
      <td>Density change → <code>rowHeight</code> de 25 → 37 → 45. Le spacer top ajuste sa hauteur, la position de scroll est <em>préservée proportionnellement</em> (premier visible reste stable).</td>
    </tr>
    <tr>
      <td><strong><code>totalCount</code> distinct de <code>rows.length</code></strong></td>
      <td>En lazy loading, on connaît le total côté serveur (<code>50_000</code>) avant de tout charger. <code>:total-count</code> dimensionne la scrollbar pour qu'elle ne saute pas quand de nouvelles pages arrivent.</td>
    </tr>
  </tbody>
</table>

## Horizontal — virtualisation des colonnes

Même principe sur l'axe X : `useVirtualColumns` calcule un `[colStart, colEnd]`
basé sur `scrollLeft` et les largeurs cumulées des colonnes. Les **colonnes
pinned (start/end) sont toujours rendues** — elles sont en `position: sticky`,
donc visibles peu importe le scroll horizontal.

<div className="ascii">
{`        ◀── scrollLeft ──▶
┌──────┬──────────────────────────────────────────────┬──────┐
│PIN(s)│ <— virtual slice (only visible columns) —→   │PIN(e)│
└──────┴──────────────────────────────────────────────┴──────┘
   ▲                                                     ▲
   ├ position: sticky; left:  0                          ├ position: sticky; right: 0
   └ always rendered                                     └ always rendered`}
</div>

<div className="callout">
  ⚠️ Quand <code>:virtual-columns</code> est actif, le mécanisme « dernière colonne flex »
  est <strong>désactivé</strong> : la slice change pendant le scroll, et faire grossir la
  dernière colonne visible créerait des sauts de largeur à chaque scroll.
</div>

## Variable height (lignes de tailles différentes)

Pour les rows qui se déplient, les groupes ou les detail rows, la hauteur n'est
pas uniforme. Le composable `useVariableHeightVirtualScroll` maintient un cache
**rolling** des hauteurs mesurées :

```ts
const v = useVariableHeightVirtualScroll({
  totalCount: () => rows.value.length,
  estimatedRowHeight: 37,         // valeur initiale, ajustée par mesure réelle
  containerRef: wrapperRef,
  scrollTop: () => wrapperRef.value?.scrollTop ?? 0,
})

// l'utilisateur expand une row → v.recomputeHeight(rowIndex, newHeight)
```

L'algorithme est en O(log N) via une `BinaryIndexedTree` (Fenwick), donc même
sur 100k lignes le `getRowOffset(i)` reste rapide.

## Pinned columns + virtualisation

Le layout est délicat : les colonnes pinned sont **en dehors** de la slice
virtuelle (toujours rendues), mais doivent rester sticky **dans le wrapper**,
pas relativement à la fenêtre. Pour ça :

- Le `.mrx-grid-row` a `min-width: max(100%, contentMinWidth)` — la rangée est
  au moins aussi large que le wrapper
- Le `.mrx-grid-cell--pinned` a `position: sticky` + un offset calculé selon le
  cumul des colonnes précédentes
- Pas de `transform` sur les ancêtres (sinon containing block trap, voir plus haut)

## Pièges connus

### 1. Le scroll ne fonctionne pas / la grid déborde son parent
Sans <code>:virtual-scroll</code> ni <code>:pagination</code>, le wrapper grandit naturellement.
S'il est dans un parent à hauteur fixe et <code>overflow:hidden</code>, le contenu est
coupé sans scroll. Wrap dans un <code>display: flex; flex-direction: column</code> →
<code>flex: 1; min-height: 0</code> sur le grid root prend le relais (déjà géré).

### 2. Pinned columns « décollent » au scroll
Probablement un <code>transform</code> ou <code>filter</code> sur un ancêtre. Vérifie via Chrome
DevTools → Computed → <em>Containing block</em>. La colonne pinned doit avoir le
wrapper comme containing block, pas un container intermédiaire.

### 3. Saut de scroll quand les données arrivent (lazy)
Pass <code>:total-count="50000"</code> dès le départ pour fixer la taille du sizer,
même si <code>rows.value.length</code> est 0 au début.
